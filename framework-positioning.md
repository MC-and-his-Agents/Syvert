# Syvert Framework Positioning

## 这是什么

Syvert 应被定义为一个面向媒体采集场景的核心框架，而不是某个平台能力的集合。

它的职责不是内置尽可能多的平台实现，而是提供一个稳定的宿主，使平台能力可以通过适配器被接入、运行、治理和审计。


## 核心定位

Syvert Core 是一个：

- 面向任务的采集执行框架
- 面向资源的运行治理框架
- 面向数据的标准化处理框架
- 面向适配器的可插拔宿主框架

更具体地说，它负责“怎么运行”，而不是“抓哪个平台”。


## 解决什么问题

Syvert Core 主要解决以下问题：

- 如何把采集请求建模为可追踪任务
- 如何为任务分配和回收运行资源
- 如何统一处理任务状态、失败分类和结果输出
- 如何把平台原始结果映射到共享数据模型
- 如何通过 API/CLI 暴露稳定能力
- 如何让不同平台的接入逻辑在同一套规则下运行

如果这些能力做得扎实，平台适配器就只是插件，而不是系统本体。


## 不是什么

Syvert Core 不应被定义为：

- 某个平台实现的抽象壳
- 一个只包含接口定义而无法运行的 SDK
- 一个脚本聚合器
- 一个默认内置高耦合平台逻辑的主仓库
- 一个以平台覆盖率为主要卖点的产品

它的价值不在“默认支持几个平台”，而在“能不能稳定承载任何平台适配器”。


## 为什么做成框架

把 Core 和 Adapter 分离，有两个直接价值：

### 1. 社区可扩展

当 Core 只提供稳定宿主能力，而平台接入通过统一 SDK 实现时，社区可以围绕适配器生态扩展。

这要求：

- Core 稳定
- 适配器接口稳定
- 接入成本可控
- 测试与兼容策略明确

社区贡献的主入口应该是 Adapter，而不是持续修改 Core。若贡献方拥有通用执行产品，它可以作为 Adapter-bound Provider 参与某些 capability 的执行，但必须通过兼容性判断证明可绑定范围。

### 2. 风险边界更清晰

当平台协议、平台反爬逻辑、平台特定请求构造都被限定在适配器层时，Core 的职责更容易被界定为通用任务与资源框架。

这不会自动消除所有风险，但会显著改善系统的责任边界和表达边界。


## 产品边界

Syvert Core 应只内置以下能力：

- 任务系统
- 任务状态机
- 任务执行器
- 资源模型与资源调度
- 共享数据模型
- 审计与日志
- API/CLI 入口
- 适配器注册与加载机制
- 适配器运行时上下文
- 统一错误模型

Syvert Core 不应内置：

- 平台特定请求代码
- 平台签名算法实现
- 平台专有字段模型
- 平台专有反爬逻辑


## Core 的最小可用定义

一个真正有价值的 Core，至少必须满足以下条件：

1. 可以提交任务
2. 可以查询任务状态
3. 可以驱动一个适配器执行
4. 可以分配和释放资源
5. 可以落盘任务结果和执行记录
6. 可以将原始结果转成统一输出
7. 可以通过 API 和 CLI 复用同一条执行链路

如果做不到这些，它只是抽象层，不是框架产品。


## 产品结构

Syvert 应以 Core、Adapter SDK、Adapters 为主结构，并以 Providers 作为 Adapter-bound 执行能力扩展层：

### 1. Core

负责：

- 任务
- 资源
- 数据
- 执行
- 审计

### 2. Adapter SDK

负责：

- 适配器接口
- 运行时上下文
- 输入输出模型
- 错误模型
- 测试规范
- 第三方 Adapter 接入路径
- Adapter / Provider 兼容性判断入口

### 3. Adapters

负责：

- 平台接入实现
- 平台响应解析
- 平台错误映射
- 平台特定策略

Adapter 可以只包含自身执行逻辑，也可以绑定 provider 执行能力。无论哪种形态，Adapter 都仍然负责 Syvert 语义边界：capability、资源需求、错误映射与 normalized result。

### 4. Providers

负责：

- 浏览器、远程执行、CLI、agent 或其他可替换执行能力
- 声明可服务的 Adapter capability 范围
- 返回 Adapter 生成最终结果所需的执行证据与原始载荷

Provider 不负责：

- 直接接入 Syvert Core
- 生成 Syvert normalized result
- 声明覆盖所有 Adapter capability
- 成为官方 provider 产品清单的默认成员


## 用户价值

对使用者而言，Syvert Core 的价值不是“立刻支持某个平台”，而是：

- 可以在统一框架里接入多个平台
- 不必为每个平台重写任务系统和资源系统
- 可以对不同适配器使用一致的治理方式
- 可以把平台接入和核心引擎解耦演进

对适配器开发者而言，价值是：

- 只需要关注平台接入本身
- 不需要自己实现完整任务框架
- 不需要自己实现资源治理和审计体系


## 商业与生态意义

从生态角度，Core 是平台生态的基础设施。

从产品角度，Core 的竞争力来自：

- 清晰边界
- 强运行时
- 好接入体验
- 稳定兼容策略

不是来自“主仓库里塞了多少平台”。

同样也不是来自“官方支持了多少 provider 产品”。OpenCLI、bb-browser、agent-browser 或其他具体 provider 只能作为验证样本或后续独立 FR 的候选，不应成为 `v1.0.0` 的主线成功标准。


## 成功信号

如果出现以下情况，说明这个定位成立：

- 新平台接入主要通过新增 adapter 完成
- 接入新平台不需要重写任务主链路
- API 和 CLI 都基于同一套任务执行逻辑
- 资源系统与平台代码保持解耦
- 社区贡献主要集中在适配器层
- Core 的版本演进不会频繁破坏现有适配器
- 第三方可以按自身能力选择 Adapter-only 或 Adapter + Provider 接入
- Provider 兼容性围绕具体 Adapter capability 判断，而不是围绕全局产品承诺判断


## 失败信号

如果出现以下情况，说明定位正在偏离：

- Core 中不断出现平台特例
- 适配器必须依赖 Core 内部私有实现细节
- 每接一个平台都要改 task/resource/data 主链路
- 平台能力成了产品卖点，Core 退化成容器
- 适配器无法独立演进，只能跟着主仓库同步改动
- 指定 provider 产品支持清单取代了 Adapter / Provider 兼容性判断
- Provider 绕过 Adapter 直接侵入 Core routing、TaskRecord 或 resource lifecycle


## 一句话定位

Syvert Core 是一个为媒体采集场景设计的任务与资源运行框架，
平台能力通过适配器接入，而不是内置在核心中。
