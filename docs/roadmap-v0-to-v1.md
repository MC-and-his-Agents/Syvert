# Syvert 路线图 v0.1.0 到 v1.0.0

## 目的

这份路线图定义了 Syvert 从第一个可运行的 Core 里程碑到稳定 `v1.0.0` 的预期演进路径。

它基于当前的方向决策：

- Syvert 是一个 Core 框架，而不是平台打包集合。
- 真实验证应来自少量参考适配器。
- 前两个参考适配器已固定：
  - 小红书
  - 抖音


## 跨版本产品目标

早期版本的目的不是最大化功能。

它的目的在于证明一件事：

**Syvert Core 能否在一个稳定的运行时契约下承载多个真实适配器，而不会把 Core 变成平台特定代码？**

`v0.x` 中的一切都应该服务于这个问题。


## 版本策略

- `v0.1.0` 到 `v0.3.0`：证明 Core 契约
- `v0.4.0` 到 `v0.6.0`：建立最小产品闭环
- `v0.7.0` 到 `v0.9.0`：稳定 SDK、资源模型和系统边界
- `v1.0.0`：宣布 Core 稳定


## v0.1.0

### 目标

创建第一个可运行的 Core。

### 必须具备

- 本地单进程运行时
- CLI 入口
- 最小任务模型
- 最小适配器接口
- 最小执行引擎
- 标准化任务输入
- 标准化任务输出
- `raw + normalized` 适配器结果契约

### 范围

只支持一个能力：

- `content_detail_by_url`

只支持同步、进程内执行：

- 提交
- 执行
- 返回结果

### 明确不在范围内

- HTTP API
- 多租户支持
- 后台队列
- 重试
- 取消
- 浏览器资源提供方
- 高级资源调度器
- 通用搜索/评论/创作者能力

### 成功标准

- 同一个 Core 可以运行：
  - 小红书适配器
  - 抖音适配器
- 两个适配器都遵循同一契约
- Core 的主执行路径不需要平台特定分支


## v0.2.0

### 目标

把可运行的 Core 变成一个契约驱动的 Core。

### 必须具备

- `InputTarget` 模型
- `CollectionPolicy` 模型
- 标准化错误模型
- 执行日志模型
- 适配器注册表
- 适配器契约测试
- 用于测试 Core 行为的假适配器

### 明确不在范围内

- 完整的资源管理 UI/API
- 浏览器资源实现
- 多租户隔离
- 分布式执行

### 成功标准

- 适配器能力可被发现
- 错误分类保持一致
- Core 测试可以在不执行真实平台的情况下验证适配器


## v0.3.0

### 目标

暴露第一个外部服务面。

### 必须具备

- HTTP API
- 任务提交端点
- 任务状态端点
- 任务结果端点
- 共享任务/结果序列化
- CLI 和 API 使用相同的执行路径

### 明确不在范围内

- 生产级认证模型
- 分布式工作队列
- 高级任务控制
- 丰富查询层

### 成功标准

- CLI 和 API 都走同一套适配器契约
- API 不绕过 Core 内部机制
- 任务在执行完成后仍可查询


## v0.4.0

### 目标

建立最小资源系统。

### 必须具备

- 账号资源模型
- 代理资源模型
- 资源获取接口
- 资源释放接口
- 资源状态跟踪
- 资源使用日志
- 最小资源状态集合：
  - `AVAILABLE`
  - `IN_USE`
  - `INVALID`

### 明确不在范围内

- 完整资源控制台
- 高级健康恢复循环
- 浏览器资源管理
- 跨租户策略复杂性

### 成功标准

- 适配器不会自行来源化执行资源
- Core 可以向适配器执行注入资源包
- 资源使用可以按任务追踪


## v0.5.0

### 目标

支持高级资源抽象，同时不绑定具体实现技术。

### 必须具备

- 浏览器类资源抽象
- 签名会话资源抽象
- 适配器资源需求声明
- Core 对资源能力匹配的支持

### 重要约束

这个版本应该增加抽象，而不是具体的浏览器实现。

它应该支持如下概念：

- 浏览器会话
- 签名会话
- 交互式会话

但不应硬编码：

- Playwright
- CDP
- Chromium

### 成功标准

- 适配器可以声明高级运行时资源需求
- Core 可以在不知道具体实现的情况下推理这些需求


## v0.6.0

### 目标

让系统具备最小可运维性。

### 必须具备

- 超时控制
- 基础重试策略
- 并发限制
- 失败分类
- 结构化日志
- 最小执行指标
- 更持久的任务/结果持久化

### 明确不在范围内

- 完整分布式调度
- 复杂策略 DSL
- 大规模租户治理

### 成功标准

- 失败是可见且可分类的
- 任务可以按预期失败，而不是静默崩溃
- 运维人员可以检查执行期间发生了什么


## v0.7.0

### 目标

稳定适配器表面。

### 必须具备

- SDK 版本标识
- 适配器兼容性声明
- 适配器能力元数据
- 适配器验证工具
- 参考适配器测试框架

### 成功标准

- 第三方适配器作者可以面向稳定契约开发
- Core 的变更不再轻易破坏适配器


## v0.8.0

### 目标

增加最小租户边界模型。

### 必须具备

- 客户端或租户身份模型
- 私有资源所有权
- 任务所有权
- 审计所有权
- 最小认证模型

### 明确不在范围内

- 完整 RBAC 系统
- 企业级组织层级
- 复杂权限图

### 成功标准

- 任务和资源边界是明确的
- 私有资源不再被视为全局共享运行时状态


## v0.9.0

### 目标

为 Core 稳定化做准备。

### 必须具备

- 契约清理
- 错误码清理
- 文档重建
- SDK 清理
- 移除已知的意外平台泄漏
- 借助两个参考适配器进行更强的端到端验证

### 成功标准

- Core 与适配器职责之间不再存在重大歧义
- 两个参考适配器不再需要 Core 中的特殊处理支持


## v1.0.0

### 目标

宣布 Syvert Core 稳定。

### v1.0.0 的要求

- Core 边界稳定
- 适配器契约稳定
- 任务模型稳定
- 资源模型稳定
- API 和 CLI 共享同一运行时路径
- 两个参考适配器运行在稳定的 Core 契约之下
- 新适配器拥有受限且有文档说明的接入路径

### v1.0.0 不代表什么

它不代表：

- 每个平台都已支持
- 每项能力都已实现
- 每种资源类型都有提供方

它代表：

- Core 已经足够稳定，可以承载一个生态


## 所有 v0.x 版本的开发规则

当真实适配器暴露出弱点时，响应应当是：

- 先判断这是否是 Core 契约问题，
- 再修改 Core，
- 除非抽象真的需要，否则不要把平台特定设计引入 Core。

这条规则至关重要。

如果没有它，参考适配器就会停止验证框架，而开始把它塑造成平台特定软件。


## 总结

这份路线图最简短的描述是：

- `v0.1.0` 证明 Core 能运行
- `v0.3.0` 证明 Core 能被使用
- `v0.6.0` 证明 Core 能被运维
- `v1.0.0` 证明 Core 值得依赖
